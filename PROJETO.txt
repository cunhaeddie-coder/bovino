  1. VISÃO GERAL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Plataforma SaaS com dois eixos principais:

  A) MARKETPLACE — compra e venda de gado no Brasil, com chat de negociação,
     alertas de demanda, cotações @arroba em tempo real e inteligência de mercado.

  B) GESTÃO PECUÁRIA — módulo completo para o produtor gerenciar sua fazenda:
     rebanho, saúde, pesagens, financeiro, estoque, equipe, pasto e curral.

  Perfis de usuário:
    - Comprador       → busca, negocia, cria alertas de demanda
    - Produtor        → anuncia, gerencia fazenda (Premium+)
    - Anunciante B2B  → veiculação de banners/campanhas
    - Admin           → painel interno com 5 roles (super/operador/ti/vendas/treinamento)

  Repositório raiz: c:\laragon\www\bovino\
  Ambiente local:   Laragon (Windows 11) + PHP 8.3.30 + MySQL

  Submódulos:
    /api    → Backend Laravel 11      (porta 8000)
    /web    → Frontend Next.js 16     (porta 3000) — app usuário final
    /admin  → Frontend Next.js 16     (porta 3001) — painel administrativo

  3. ARQUITETURA DA API
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  VERSIONAMENTO
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ Todas as rotas públicas e autenticadas usam prefixo /api/v1/            │
  │ Rotas admin usam /api/admin (prefixo fixo em bootstrap/app.php)        │
  │ Webhook MercadoPago: /api/webhook/mercadopago (sem v1 — URL externa)   │
  └─────────────────────────────────────────────────────────────────────────┘

  CAMADAS
  ├── Controllers     — "magros": recebem FormRequest, chamam lógica, retornam JSON
  ├── FormRequests    — validação e autorização (app/Http/Requests/)
  ├── Services        — lógica de negócio reutilizável (app/Services/)
  ├── Events          — eventos de domínio (app/Events/)
  ├── Listeners       — reações aos eventos (app/Listeners/)
  ├── Jobs            — processamento assíncrono em fila (app/Jobs/)
  └── Models          — Eloquent com casts, relations, accessors

  PADRÕES ADOTADOS
  - FormRequests com prepareForValidation() para normalizar CPF/celular
  - withValidator() para regras de negócio cruzadas (unicidade CPF, negociação ativa)
  - Events disparados após criação: NovoAnuncio, NovaNegociacao, TransacaoConfirmada
  - Idempotência no webhook: checar gateway_id antes de atualizar assinatura
  - Offline sync: campo coletado_em em pesagens e eventos_campo

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Três identidades distintas via Laravel Sanctum (Bearer token):

  ┌────────────────┬──────────────┬─────────────────────────────────────────┐
  │ Guard          │ Modelo       │ Login endpoint                          │
  ├────────────────┼──────────────┼─────────────────────────────────────────┤
  │ sanctum        │ User         │ POST /api/v1/auth/login                 │
  │ sanctum        │ Anunciante   │ POST /api/v1/anunciante/login           │
  │ sanctum        │ Admin        │ POST /api/admin/login                   │
  └────────────────┴──────────────┴─────────────────────────────────────────┘

  - Token de User expira em 15 min → renovado via POST /api/v1/auth/refresh
  - Token de Admin sem expiração (revogado no logout)
  - Tokens armazenados em localStorage
  - /auth/me retorna assinatura_ativa completa no payload

  Verificação de acesso:
  - Celular OTP obrigatório no cadastro
  - users.plano = "free" | "premium" (campo simplificado)
  - Plano real em assinaturas.plano_id → verificado via temPlano(user) em auth.ts

  12. SISTEMA DE MATCHMAKING DE ALERTAS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Fluxo automático ao criar anúncio:
  1. POST /v1/anuncios → AnuncioController::store()
  2. NovoAnuncio::dispatch($anuncio) → Event
  3. MatchingAlertasDemanda::handle() → Listener
     a. Filtra alertas_demanda ativos compatíveis:
        - raca (nulo = qualquer)
        - sexo (nulo = qualquer, misto = ambos)
        - estado dentro do JSON array estados (vazio = todos)
        - faixa de peso min/max
        - exclui o próprio vendedor
     b. Insere notificações em lote na tabela notificacoes
     c. Atualiza alertas_demanda.ultimo_alerta_at

  Registro no AppServiceProvider:
  Event::listen(NovoAnuncio::class, MatchingAlertasDemanda::class)

  13. APP CURRAL — OFFLINE SYNC
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Fluxo offline-first:
  1. GET /gestao/curral/dados-offline → retorna animais + lotes, salva em localStorage
  2. App coleta pesagens e eventos SEM internet, armazena em:
     curral_dados, curral_pesagens, curral_eventos, curral_sessao
  3. Quando online: POST /gestao/curral/sessoes → cria sessão
  4. POST /gestao/curral/sessoes/{id}/sincronizar
     → envia { pesagens: [..., coletado_em], eventos: [..., coletado_em] }
     → DB::transaction: cria Pesagem, atualiza peso_atual no Rebanho, cria EventoCampo
     → marca sessao.sincronizado_em = now()

  Campo coletado_em:
  - Registra quando o dado foi coletado no campo (pode ser dias antes do sync)
  - created_at = timestamp do sync (quando entrou no banco)
  - coletado_em = timestamp real de coleta (enviado pelo app)
  - Fallback: se não enviado, usa data_pesagem ou data_evento

  17. DECISÕES TÉCNICAS IMPORTANTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  1. CONFLITO: tabela "animais" já existia para o marketplace.
     Gestão pecuária usa tabela "rebanho" (Model: Rebanho, $table='rebanho').
     Nunca renomear — quebra o marketplace.

  2. CONFLITO: tabela "negociacoes" já existia para o chat comprador-vendedor.
     Negócios confirmados ficam em "transacoes" (Model: Transacao).

  3. ROTAS: todas em /api/v1/* exceto:
     - /api/webhook/mercadopago — URL cadastrada externamente no gateway
     - /api/admin/* — prefixo fixo em bootstrap/app.php (arquivo admin.php separado)
     - /api/dev/otp/* — helper local apenas

  4. TAILWIND CSS 4: usar bg-linear-to-r (deprecated: bg-gradient-to-r).

  5. useSearchParams() no Next.js 15/16 exige <Suspense> no componente pai.

  6. users.plano é campo simplificado ("free"|"premium"). Plano real em
     assinaturas.plano_id. Ao ativar qualquer assinatura, atualizar AMBOS.

  7. Token User expira em 15 min. O interceptor em web/lib/api.ts faz refresh
     automático via POST /v1/auth/refresh no 401.

  8. buscas_log e movimentacoes_estoque: $timestamps=false, só created_at.

  9. Assinatura é morphTo (User | Anunciante). Usar get_class($assinante).

  10. PAGAMENTO IDEMPOTÊNCIA: MercadoPagoService verifica se gateway_id já
      existe com mesmo status antes de processar. Evita dupla extensão de
      assinatura em reenvio de webhook.

  11. FILA DE WEBHOOK: WebhookController retorna 200 imediatamente e despacha
      ProcessarWebhookMercadoPago (Job, tries=3, backoff=30s). Para rodar:
      php artisan queue:work
      Em testes: QUEUE_CONNECTION=sync no .env executa na mesma requisição.

  12. FORMREQUESTS: normalizar CPF e celular em prepareForValidation() antes
      das regras de validação. Regras de unicidade e negócio em withValidator().

  13. EVENTOS DE DOMÍNIO: AppServiceProvider.boot() registra os listeners.
      Não usar EventServiceProvider (removido no Laravel 11).

  14. IA GESTOR: chamada direta à Anthropic REST API via Http::post() do Laravel.
      Sem SDK PHP. Modelo: claude-haiku-4-5-20251001. Contexto da fazenda
      (total animais, alertas saúde, eventos abertos) injetado no system prompt.

  15. MIGRAÇÃO 300003 (lotes_gestao): tabela rebanho_lote (pivot) está
      NESTA migration, não na 300002, pois depende de FK para lotes_gestao.
